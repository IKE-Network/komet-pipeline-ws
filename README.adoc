= Komet Pipeline Workspace
:toc:
:toc-placement!:

Workspace aggregator for the Komet application stack: knowledge
architecture primitives (tinkar-core), persistent storage (rocks-kb),
knowledge editor (komet), and desktop distribution (komet-desktop).

toc::[]

== How This Differs from tinkar-komet-desktop-ws

This workspace supersedes
https://github.com/kec/tinkar-komet-desktop-ws[tinkar-komet-desktop-ws]
with several improvements:

Updated tooling::
Uses `ike-maven-plugin` v47 (was v29). Benefits from all
`InitWorkspaceMojo` improvements: automatic Maven wrapper installation,
`.mvn/jvm.config` creation, and post-checkout hook setup per component.

Improved Syncthing integration::
The shared Syncthing ignore rules (`stignore-shared`) now use the
`(?d)` prefix on directory-level patterns (`.git/`, `target/`, etc.).
This tells Syncthing to delete ignored files when their parent directory
is removed on the remote machine, preventing the "directory has been
deleted but contains ignored files" errors that previously required
manual cleanup. See <<stignore-changes>>.

Workspace branching model::
The workspace itself is versioned in Git, enabling developers to branch
the workspace configuration. A workspace branch captures which
component branches and versions are in use, so developers can check out
each other's workspace setups without impacting the aggregated projects.
This supports parallel experimentation: one developer can work on a
feature branch that requires coordinated changes across multiple
components, while another developer's workspace remains on `main`.

Incremental evolution::
Demonstrates evolution of workspace capabilities without directly
modifying the aggregated projects. The component repos (tinkar-core,
rocks-kb, komet, komet-desktop) are identical -- only the workspace
tooling and conventions advance.

IntelliJ fast incremental builds::
The workspace aggregator has `root="true"` in its POM, marking the
Maven 4.1.0 reactor boundary. Component sub-repos do _not_ set
`root="true"`, so IntelliJ can open any component individually as a
standalone project and perform fast incremental builds without
resolving the full workspace reactor.

== Bootstrap

[source,bash]
----
git clone https://github.com/IKE-Network/komet-pipeline-ws.git
cd komet-pipeline-ws
mvn ike:init        # <1>
mvn clean install   # <2>
----
<1> Clones all component repos in dependency order; installs Maven
    wrapper and JVM config per component.
<2> Builds the full stack. Use `-pl tinkar-core -am` to build a
    single component and its dependencies.

== Environment Setup

=== 1. Run workspace upgrade

[source,bash]
----
mvn ike:ws-upgrade
----

This brings the workspace in line with current conventions:
global gitignore entries, plugin version, stignore flags, and
other incremental improvements. Idempotent — safe to run repeatedly.

=== 2. Run health check (Syncthing users only)

[source,bash]
----
~/ike-dev/scripts/health-check.sh
----

Verify all checks pass. Address any warnings (typically stale
`.sync-conflict-*` files).

== Optional: Syncthing Multi-Machine Setup

Syncthing enables real-time file synchronization between a developer's
machines (e.g., desktop and laptop). This is **optional** — it is only
needed if you develop on two machines and want uncommitted changes to
propagate without committing to git. Skip this section if you work on
a single machine.

See link:SPLIT-BRAIN.adoc[SPLIT-BRAIN.adoc] for the full multi-machine
architecture, prevention, and recovery guide.

=== Install Syncthing

macOS::
Install https://github.com/syncthing/syncthing-macos/releases[Syncthing
for macOS] (v2.0.14+). The macOS app wraps the Syncthing daemon with a
menu-bar icon and automatic startup.

Windows::
Install https://github.com/canton7/SyncTrayzor/releases[SyncTrayzor]
(recommended) or the
https://github.com/syncthing/syncthing/releases[standalone Syncthing
binary]. SyncTrayzor provides a system-tray wrapper with auto-start and
a built-in web UI.

Linux::
Install via your package manager (`apt install syncthing`,
`dnf install syncthing`, etc.) or download the
https://github.com/syncthing/syncthing/releases[binary]. Enable the
systemd user service: `systemctl --user enable --now syncthing`.

Accept the shared `ike-dev` folder from the other machine. Set the
local path to `~/ike-dev` (macOS/Linux) or `%USERPROFILE%\ike-dev`
(Windows).

=== Configure ignore rules

Ensure `.stignore` exists at the root of the shared folder:

[source,bash]
----
# macOS / Linux:
echo '#include stignore-shared' > ~/ike-dev/.stignore

# Windows (PowerShell):
Set-Content "$env:USERPROFILE\ike-dev\.stignore" '#include stignore-shared'
----

[[stignore-changes]]
The shared ignore rules use `(?d)` prefixes on directory patterns
(`.git/`, `target/`). The `(?d)` flag tells Syncthing to delete
ignored files when their parent directory is removed on the remote
machine, preventing persistent "out of sync" errors. Run
`mvn ike:ws-upgrade` to apply these flags if they are missing.

=== Install git hooks and watcher

[source,bash]
----
~/ike-dev/scripts/setup-hooks.sh      # Global post-checkout/post-merge hooks
~/ike-dev/scripts/install-watcher.sh  # launchd agent for auto git-init
----

The hooks generate `_git-init.sh` scripts when you clone or checkout
repos under `ike-dev/`. The watcher automatically runs these scripts
when new repos arrive via Syncthing on the other machine.

== Polyglot Flags

When `mvn ike:init` clones components, it creates a
`.mvn/jvm.config` file in each component with:

----
-Dpolyglotimpl.AttachLibraryFailureAction=ignore
----

This suppresses GraalVM TruffleAttach warnings when building on
standard (non-GraalVM) JDKs. The IKE build stack uses GraalVM
polyglot libraries (via `maven-skin-tools` for site rendering), and
without this flag, non-GraalVM JDKs produce harmless but noisy
warnings during the build.

The flag must live in `.mvn/jvm.config` because Maven reads it before
POM parsing -- it cannot be inherited from a parent POM or set as a
Maven property.

== Split-Brain Prevention

Working on the same files from two machines simultaneously creates
Syncthing conflict files and potential confusion. See
link:SPLIT-BRAIN.adoc[SPLIT-BRAIN.adoc] for the full prevention,
detection, and resolution guide.

== Workspace Commands

All `ike:` goals are standard Maven goals and appear in the Maven
tool window of IntelliJ IDEA (under _Plugins > ike_). You can
double-click any goal to run it -- no command-line memorization
required.

=== Setup and Status

[cols="1,3", options="header"]
|===
| Goal | Description

| `ike:init`
| Clone all components from `workspace.yaml` in dependency order.
  Installs Maven wrapper (for `defaults.maven-version`), creates
  `.mvn/jvm.config`, and sets up post-checkout hooks per component.
  Pass `-Dgroup=<name>` to restrict to a group or single component.

| `ike:setup`
| Install VCS bridge git hooks to `~/.git-hooks/` for Syncthing
  coordination (pre-commit, post-commit, pre-push). Does not
  overwrite existing hooks unless `-Dforce=true`.

| `ike:status`
| Show branch, short SHA, and clean/dirty status for each component.
  Pass `-Dgroup=<name>` to restrict scope.

| `ike:dashboard`
| Morning standup command: manifest health check, component status,
  and cascade analysis (what needs rebuild) in one pass.

| `ike:pull`
| Run `git pull --rebase` across all components in dependency order.
  Pass `-Dgroup=<name>` to restrict scope.

| `ike:graph`
| Print the workspace dependency graph. Use `-Dformat=dot` for
  Graphviz DOT output (default: text).

| `ike:verify`
| Verify manifest consistency: dangling dependencies, missing
  components, cycle detection.

| `ike:cascade`
| Show which components are affected by changes in dirty components
  (breadth-first dependency traversal).

| `ike:stignore`
| Generate deterministic `.stignore` files for the workspace and
  each component. Ensures all machines converge on the same
  Syncthing ignore configuration.

| `ike:help`
| List all available `ike:` goals with descriptions.
|===

=== Feature Branching

Feature branches are coordinated across workspace components.
`feature-start` creates `feature/<name>` branches with
branch-qualified SNAPSHOT versions (e.g.,
`1.2.0-my-feature-SNAPSHOT`), so artifacts from different features
coexist safely in `~/.m2/repository`.

[cols="1,3", options="header"]
|===
| Goal | Description

| `ike:feature-start`
| Start a coordinated feature branch. Prompts for the feature name
  if `-Dfeature=<name>` is not provided. Creates branches, sets
  branch-qualified versions, and updates `workspace.yaml`. Pass
  `-Dgroup=<name>` to branch a subset of components.

| `ike:feature-start-dry-run`
| Preview what `feature-start` would do without executing.

| `ike:feature-finish-squash`
| **Default and recommended.** Squash-merge the feature branch to
  the target (default: `main`). All feature commits become one clean
  commit. Feature branch is deleted after merge (squash creates
  divergent history). Prompts for a commit message if `-Dmessage`
  is not provided.

| `ike:feature-finish-merge`
| No-fast-forward merge preserving full feature history. Feature
  branch is kept by default -- it can continue to receive work and
  be merged again later. Good for long-lived feature branches with
  periodic intermediate merges.

| `ike:feature-finish-rebase`
| Rebase feature onto target, then fast-forward merge. Produces
  linear history without a merge commit. Caution: rebasing rewrites
  history -- other machines must run `ike:sync` afterward.
|===

All feature-finish goals accept `-Dfeature=<name>` (prompted if
omitted), `-DtargetBranch=<branch>` (default: `main`),
`-DkeepBranch=true|false`, and `-DdryRun=true`.

=== Syncthing Synchronization

[cols="1,3", options="header"]
|===
| Goal | Description

| `ike:ws-sync`
| Synchronize `workspace.yaml` with actual git branches.
  `-Dfrom=repos` (default): update YAML from checked-out branches.
  `-Dfrom=manifest`: switch repos to match YAML declarations.

| `ike:sync`
| Reconcile git state after Syncthing delivers changes from another
  machine. Reads VCS state files, fetches, switches branches, and
  soft-resets HEAD. Working tree is untouched -- Syncthing already
  made it correct.
|===

=== Checkpointing and Release

[cols="1,3", options="header"]
|===
| Goal | Description

| `ike:ws-checkpoint`
| Create a workspace checkpoint: build, tag, and deploy every
  component in dependency order, then record coordinates in a YAML
  file under `checkpoints/`. The checkpoint name is auto-derived
  from the workspace branch and a compact UTC timestamp (e.g.,
  `main-20260402-143022`). Override with `-Dname=<label>` when a
  custom name is needed. Pass `-DskipVerify=true` to skip tests.

| `ike:ws-checkpoint-dry-run`
| Preview checkpoint without writing files or creating tags.

| `ike:ws-release`
| Release all dirty components in dependency order. Auto-detects
  which components have commits since their last release tag.
  After each release, updates parent/BOM versions in downstream
  components. Pass `-DdryRun=true` to preview.

| `ike:release`
| Full release for a single component: build, deploy, tag, merge
  back to main, and bump to next SNAPSHOT. All local git work
  completes before external actions (deploy, push).
|===
